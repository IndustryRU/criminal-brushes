# Technical Specification

## 1. Назначение системы

Разработать web и mobile e-commerce приложение для бренда `Criminal Brushes` на стеке `Flutter + Supabase`.

Платформы:

- Web.
- Android для RuStore.
- Android для Google Play.
- iOS для App Store.

## 2. Пользовательские роли

### Guest

- Просмотр главной.
- Просмотр каталога.
- Просмотр карточки товара.
- Добавление товаров в локальную корзину.
- Переход к checkout.

### Customer

- Все возможности гостя.
- Авторизация.
- Сохранение адресов.
- История заказов.
- Избранное.
- Получение уведомлений.
- Оставление отзывов после покупки.

### Manager

- Просмотр заказов.
- Изменение статусов заказов.
- Просмотр клиентов без доступа к чувствительным данным.
- Управление остатками.

### Admin

- Полный доступ к каталогу, заказам, промокодам, контенту и ролям.
- Просмотр audit log.
- Управление публикацией контентных блоков.

## 3. Основные модули

### Главная

Функции:

- Hero-блок с главным drop-продуктом.
- Таймер до конца drop или промо.
- Быстрое добавление hero-товара в корзину.
- Блок преимуществ.
- Коллекция товаров.
- Промо-баннер.
- FAQ.
- Footer с юридическими ссылками.

Состояния:

- Данные загружены.
- Ошибка загрузки контента.
- Нет активного drop.

### Каталог

Функции:

- Список товаров.
- Фильтр по коллекции.
- Фильтр по цвету.
- Фильтр по наличию.
- Сортировка по популярности, новизне, цене.
- Карточка товара с ценой, рейтингом и быстрым добавлением.

### Карточка товара

Функции:

- Галерея изображений.
- Название, коллекция, цвет.
- Цена.
- Рейтинг и отзывы.
- Выбор цвета/SKU.
- Остаток.
- Характеристики.
- Блок безопасности и гарантии.
- Добавление в корзину.
- Избранное.

### Корзина

Функции:

- Список товаров.
- Изменение количества.
- Удаление позиции.
- Промокод.
- Предварительная стоимость доставки.
- Итого.
- Переход к checkout.

### Checkout

Шаги:

1. Контакты.
2. Адрес доставки.
3. Метод доставки.
4. Оплата.
5. Подтверждение заказа.

Требования:

- Заказ создается до перехода на оплату.
- После оплаты статус обновляется через webhook.
- Все платежные проверки проходят на backend/Edge Function.
- Пользователь видит понятный экран успеха или ошибки.

### Личный кабинет

Функции:

- Профиль.
- Адреса.
- История заказов.
- Статус заказа.
- Избранное.
- Настройки уведомлений.
- Выход.

### Админка

Функции:

- CRUD товаров.
- CRUD вариантов товара.
- Загрузка изображений.
- Управление коллекциями.
- Управление заказами.
- Управление промокодами.
- Управление контентными блоками.
- Просмотр audit log.

## 4. Авторизация

Минимум:

- Email + password.
- Magic link.
- OAuth можно добавить позже.

Требования:

- Supabase Auth.
- Профиль пользователя в таблице `profiles`.
- Роли хранятся отдельно и проверяются через RLS/Edge Functions.
- Service role key не используется в клиентском приложении.

## 5. Платежи

Решение выбирается по рынку и юридическому лицу. Кандидаты:

- YooKassa.
- CloudPayments.
- Robokassa.
- ЮMoney.
- RuStore Pay для релевантных Android-сценариев.

Требования:

- Клиент не доверенный источник статуса оплаты.
- Истина о платеже приходит через webhook платежного провайдера.
- Webhook валидируется на сервере.
- В таблице `payments` хранится внешний ID платежа, статус, сумма, валюта, timestamps.
- Возвраты должны быть предусмотрены в модели данных, даже если UI появится позже.

## 6. Доставка

MVP:

- Самостоятельно заданные методы доставки в админке.
- Стоимость по простым правилам: фиксированная, бесплатная от суммы, ручной режим.

После MVP:

- СДЭК.
- Boxberry.
- Почта России.
- Яндекс Доставка.

## 7. Уведомления

Каналы:

- Email.
- Push.
- SMS опционально.

События:

- Заказ создан.
- Оплата успешна.
- Оплата не прошла.
- Заказ передан в доставку.
- Заказ доставлен.
- Промо/drop стартовал.

События:

- `app_open`

Инструменты:

- AppMetrica для российского рынка.

- PostHog для web/product analytics.
- Sentry для ошибок.

## 9. Безопасность

Обязательные требования:

- Row Level Security включена на всех пользовательских таблицах.
- Админские действия только через проверенные роли.
- Платежные операции только через Edge Functions.
- Все переменные окружения через secret storage.
- Логи не содержат токены, карты, пароли, персональные документы.
- Rate limiting для публичных Edge Functions.
- Backup Supabase перед крупными миграциями.

## 10. Нефункциональные требования

- Web Lighthouse Performance: целевой минимум 80+ на MVP.
- Crash-free sessions: 99%+.
- Поддержка Android 8+ как нижняя граница обсуждаема, но target SDK должен соответствовать требованиям магазина.
- iOS minimum version определить перед разработкой, рекомендовано не уходить слишком низко.
- Все ключевые экраны адаптированы под mobile и desktop.
- Доступность: контраст, размеры tap targets, alt/semantics для важных элементов.
